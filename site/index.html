openharness — the 90-second story
01

Agents are everywhere. Control is nowhere.

Every team wires its own prompts, keys, and tools into scattered agent tools. The company gets no shared standard, no view of what agents did, and secrets sprayed across laptops. OpenHarness centralizes the definition of how agents work — while execution stays local and the whole stack stays yours.

The model is becoming a commodity. The harness is the moat.
02

One definition → two frontends, fully governed

A HarnessDefinition is a folder: a system prompt, mandatory skills, MCP tools, a policy, branding, and which credentials to use. It becomes both surfaces from the same core.

TUI

For your engineers

A branded terminal harness over a real agent core — the full-power surface for technical staff.

Desktop

For everyone else

A native cross-platform app (Windows · Linux · macOS). Real chat UI, no terminal required.

03

Governance without losing control

Enforcement lives in-process, at the seam every tool call passes through — it can't be prompt-jailbroken or routed around. The org defines it; the harness applies it, offline.

MCP

Tools you control

MCP servers bridged in as mcp__server__tool. Mandatory ones must connect. Secrets referenced by name — never baked into the shipped app.

Policy

Deny by default

Claude-Code-style rules: allow / deny / ask, plus secret redaction on the way in and out. A denied action shows a branded block, not a leak.

Audit

What agents did

Hash-chained JSONL of external calls only — never prompts. The server retains the authoritative, continuity-checked record.

Bundles

Signed config

Definitions are ed25519-signed. The app boots pinned to a verified one and refuses anything unsigned, tampered, or rolled back.

04

The seal: flip one byte, it refuses

openharness build turns a definition into a branded, signed desktop app — each brand fully isolated. The app boots pinned to its signed configuration.

  • Two brands, one substrate — distinct identity, isolated credentials & audit.
  • The org's private key never enters the artifact (there's a test that greps for it).
  • Tamper the config → the app opens to an integrity-refusal screen, not your agent.
integrity check

$ openharness bundle verify acme.ohbundle

bundle OK — acme-fintech@0.1.0 verified

 

# an attacker flips one byte…

05

Bring your own model

API keys, gateway subscriptions (OpenCode Go), or a local runtime — plug in whatever, switch mid-session, rotate multiple org accounts on rate-limit. The bet: models commoditize, so we stay neutral on them and invest in the layer that lasts.

06

Ship a branded app in three commands

Self-hosted, no SaaS in the loop. Open the box and build.

01 · sign

Mint your org key

openharness keygen — the private key stays with you; the public key rides in the app.

02 · build

Brand + sign

openharness build acme/ → a signed, isolated desktop app with your policy baked in.

03 · ship

Distribute

Push the signed definition; installed apps re-fetch, verify, and refuse anything stale or forged.
